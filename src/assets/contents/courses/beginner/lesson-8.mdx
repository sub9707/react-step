## 번들러(Bundler)란?

먼저 이런 도구들이 왜 필요한지부터 이해해봅시다.

<Image
  src="http://subdevpi.duckdns.org:3000/file/react-step/content/beginner/8/bundler.png"
  alt="번들러 개념"
  width="full"
  caption="여러 개의 파일을 하나로 합쳐서 브라우저가 이해할 수 있게 만드는 것이 번들러의 역할입니다"
/>

React로 개발할 때 우리는 여러 개의 파일을 만들게 됩니다
- JSX 파일 (React 컴포넌트)
- CSS 파일 (스타일)
- JavaScript 파일 (로직)
- 이미지, 폰트 등 자산 파일

하지만 브라우저는 JSX를 바로 이해할 수 없습니다. 또한 수십, 수백 개의 파일을 각각 다운받는 것은 비효율적이죠.
<br/>
<HighlightWord>번들러(Bundler)</HighlightWord>는 이런 문제를 해결해주는 도구입니다

번들러는 여러 개의 파일을 분석해서, 의존성에 따라 <HighlightWord>하나의 파일 또는 최적화된 여러 파일로 묶어줍니다</HighlightWord>.

React처럼 모듈 기반으로 개발할 때는 컴포넌트마다 파일이 분리되어 있기 때문에, 이를 그대로 브라우저에 전달하면 효율이 떨어집니다.

이런 개별 파일들을 분석하여 어떤 파일이 어떤 파일을 사용하는지(의존성)를 파악하고, 필요한 순서대로 합쳐 하나의 실행 가능한 파일을 만들어냅니다.
<br/>
이 과정에서 보통 다음과 같은 작업이 함께 이루어집니다

- JSX → JS로 변환 (Transpiling)
- ES6 모듈 → 브라우저가 이해할 수 있는 형식으로 변환
- 코드 압축 (Minification)
- 사용되지 않는 코드 제거 (Tree-shaking)
- CSS, 이미지 등의 자산도 함께 포함 (Asset bundling)
<br/>

이 모든 작업은 넓은 의미에서 <HighlightWord>‘컴파일’</HighlightWord>이라고 부르기도 하지만,

더 정확히는 <HighlightWord>빌드(Build)</HighlightWord> 과정에 해당합니다.

빌드 과정 안에는 컴파일(코드 변환)뿐 아니라 번들링, 압축, 최적화 등이 모두 포함됩니다.

보통의 컴파일이라 하면, 사람이 작성한 소스코드를 기계가 이해할 수 있도록 변환하는 과정일 일컫지만,

여기서는 더 정확히 '빌드 과정'을 이릅니다.

JSX 코드를 자바스크립트 코드로 변환하고, ES6 문법을 ES5로 바꾼다던지, 코드를 압축, 난독화, 정리를 하는 등의 이 과정을 컴파일이라 하죠.

<br/>

이 빌드 과정을 거치면 모듈 기반 코드가 브라우저에서 빠르고 효율적으로 실행될 수 있습니다.

대표적인 번들러로는 **Webpack, Vite, Parcel** 등이 있으며, Create React App(CRA)에서도 내부적으로 Webpack을 사용하고 있습니다.

## Create React App (CRA)

<Image
  src="http://subdevpi.duckdns.org:3000/file/react-step/content/beginner/8/screencast.svg"
  alt="Create React App"
  width="two-thirds"
  caption="이미지 출처: [https://create-react-app.dev/docs/getting-started/]"
/>

### CRA란?

Create React App은 <HighlightWord>React 팀에서 공식적으로 제공하는 프로젝트 생성 도구</HighlightWord>입니다.

복잡한 설정 없이 한 번의 명령어로 React 프로젝트를 시작할 수 있게 해줍니다.

```bash
npx create-react-app my-app
cd my-app
npm start
```

이 세 줄로 바로 React 개발을 시작할 수 있습니다.

### CRA의 장점

<CalloutBox type="success" title="CRA의 장점">
**간단한 사전 설정**: 복잡한 설정 없이 바로 시작 가능

**React 공식 지원**: React 팀에서 직접 관리하고 업데이트

**안정성**: 오랫동안 검증된 도구로 안정적

**넓은 커뮤니티와 풍부한 자료**: 관련 튜토리얼과 문서가 많음
</CalloutBox>

### CRA의 단점

<CalloutBox type="warning" title="CRA의 단점">
**느린 시작 속도**: 개발 서버 시작이 느림 (30초~1분)

**무거운 번들**: 기본 프로젝트도 용량이 큼

**제한된 커스터마이징**: 설정 변경이 어려움

**느린 빌드**: 프로젝트가 커질수록 빌드 시간 증가

**유지보수 이슈**: 2023년부터 업데이트가 거의 중단됨
</CalloutBox>

<HighlightWord>느린 시작 속도?</HighlightWord>

→ CRA는 Webpack 기반으로 무거운 설정과 의존성을 초기부터 로딩합니다. 

개발 서버를 실행할 때 모든 설정을 해석하고 준비하는 데 시간이 오래 걸립니다.

<HighlightWord>무거운 번들?</HighlightWord>

→ 기본적으로 많은 polyfill과 설정이 포함되어 있어서, 아무것도 안 해도 초기 번들 크기가 큽니다. 

불필요한 코드도 함께 포함되는 경우가 많습니다.

<HighlightWord>제한된 커스터마이징?</HighlightWord>

→ CRA는 설정 파일(예: Webpack config)을 감추고 있어, 특별한 설정을 위해선 eject 명령을 사용해야 하는데, 

이 경우 설정이 복잡해지고 되돌리기 어렵습니다.

<HighlightWord>느린 빌드?</HighlightWord>

→ Webpack의 복잡한 트리 처리와 전체 파일 분석 방식 때문에, 프로젝트가 커질수록 빌드 시간이 급격히 느려집니다.

### CRA 프로젝트 구조

```bash
my-app/
├── public/
│   ├── index.html
│   └── favicon.ico
├── src/
│   ├── App.js
│   ├── App.css
│   ├── index.js
│   └── index.css
├── package.json
└── README.md
```

이 기본 구조는 React 앱을 구성하는 데 필요한 최소한의 틀을 제공하며, 여기에 기능이 추가될수록 더 많은 폴더와 파일이 생성되게 됩니다.

## Vite

<Image
  src="http://subdevpi.duckdns.org:3000/file/react-step/content/beginner/8/vite.jpg"
  alt="Vite"
  width="two-thirds"
  caption="번개처럼 빠른 개발 경험을 제공하는 차세대 번들러"
/>

### Vite란?

Vite(비트)는 <HighlightWord>Vue.js 창시자가 만든 차세대 프론트엔드 빌드 도구</HighlightWord>입니다.

"Vite"는 프랑스어로 "빠르다"는 뜻이며, 이름 그대로 정말 빠른 개발 경험을 제공합니다.

아래와 같이 프로젝트를 시작할 수 있습니다.

```bash
npm create vite@latest my-react-app -- --template react
cd my-react-app
npm install
npm run dev
```

### Vite의 장점

<CalloutBox type="success" title="Vite의 장점">
**초고속 시작**: 개발 서버가 1-2초 만에 시작

**빠른 HMR**: 코드 변경사항이 즉시 반영 (Hot Module Replacement)

**작은 번들**: 불필요한 코드 없이 깔끔

**유연한 설정**: 쉽게 커스터마이징 가능

**최신 기술**: ES modules, esbuild 등 최신 기술 활용

**다양한 프레임워크**: React, Vue, Svelte 등 모두 지원
</CalloutBox>

  src="http://subdevpi.duckdns.org:3000/file/react-step/content/beginner/8/vite-frameworks.png"
  alt="Vite"
  width="full"
  caption="vite는 다양한 프레임워크와 툴을 지원한다"
/>
<br/>

### Vite가 빠른 이유

<Image
  src="http://subdevpi.duckdns.org:3000/file/react-step/content/beginner/8/esbuild_speed.png"
  alt="Vite 속도 비교"
  width="two-thirds"
  caption="Vite는 ES modules를 활용합니다. ES modules은 Javascript 번들러 중 가장 빠른 속도를 자랑합니다."
/>

Vite가 빠른 핵심 이유는 <HighlightWord>ES modules</HighlightWord>를 활용하기 때문입니다:

Webpack은 모든 파일을 한 번에 번들링하고 서버를 재시작해야 하지만,

Vite는 ES Module을 사용해 <HighlightWord>필요한 파일만 실시간으로 불러오기 때문</HighlightWord>에 더 빠릅니다.
<br/>
또한, Vite는 빌드 도구로 esbuild를 사용해 설정 없이도 <HighlightWord>초고속 트랜스파일링</HighlightWord>을 지원합니다.

즉, 초기 실행 속도와 수정 반영 속도 모두 Webpack보다 훨씬 빠릅니다.

<br/>
### Vite의 단점

<CalloutBox type="warning" title="Vite의 단점">
**상대적으로 새로움**: CRA에 비해 역사가 짧음

**학습 곡선**: 고급 설정에는 Rollup에 관한 지식 필요함

**호환성**: 아주 오래된 브라우저는 지원하지 않을 수도 있음
</CalloutBox>

## Parcel

<Image
  src="http://subdevpi.duckdns.org:3000/file/react-step/content/beginner/8/parcel.png"
  alt="Parcel"
  width="full"
  caption="설정이 전혀 필요 없는 제로 설정 번들러"
/>

### Parcel이란?

Parcel은 <HighlightWord>"제로 설정" 번들러</HighlightWord>로 유명합니다.

설정 파일 하나 없이도 대부분의 웹 프로젝트를 자동으로 처리해줍니다.

```bash
npm install -g parcel-bundler
# 또는 프로젝트별로
npm install --save-dev parcel

# HTML 파일만 있으면 바로 시작
parcel index.html
```

### Parcel의 장점

<CalloutBox type="success" title="Parcel의 장점">
**완전 제로 설정**: 설정 파일이 전혀 필요 없음

**자동 감지**: 파일 형식을 자동으로 인식하고 처리

**빠른 빌드**: 멀티코어를 활용한 병렬 처리

**핫 리로딩**: 코드 상태를 유지하며 즉시 반영

**간단함**: 복잡한 개념 없이 바로 사용 가능
</CalloutBox>

### Parcel의 단점

<CalloutBox type="warning" title="Parcel의 단점">
**제한된 커스터마이징**: 세세한 설정 변경 어려움

**작은 생태계**: 플러그인과 확장 도구가 적음

**복잡한 프로젝트**: 대규모 프로젝트에서는 한계

**디버깅 어려움**: 자동화로 인한 문제 추적 어려움
</CalloutBox>

## 언제 어떤 도구를 선택해야 할까?

### Create React App을 선택

<CalloutBox type="info" title="CRA를 선택하는 경우">
**React 입문자**: 처음 React를 배우는 경우

**팀 프로젝트**: 팀 프로젝트 진행 시 표준화된 환경이 필요한 경우

**복잡한 설정을 피할 때**: 번들러 설정에 시간 쏟는 것을 피할 수 있음
</CalloutBox>

### Vite를 선택

<CalloutBox type="info" title="Vite를 선택하는 경우">
**개발 속도 중시**: 빠른 피드백이 중요한 경우

**최신 기술**: 최신 도구와 기법을 사용하고 싶은 경우

**커스터마이징**: 프로젝트 설정을 세밀하게 조정하고 싶은 경우

**성능 최적화**: 번들 크기와 로딩 속도가 중요한 경우
</CalloutBox>

### Parcel을 선택

<CalloutBox type="info" title="Parcel을 선택하는 경우">
**빠른 프로토타이핑**: 아이디어를 빠르게 테스트하고 싶은 경우

**설정 없는 개발**: 번들러 설정에 전혀 신경 쓰고 싶지 않은 경우

**소규모 프로젝트**: 간단한 개인 프로젝트나 실험

**멀티 프레임워크**: React 외에도 다양한 기술을 섞어 쓸 때

**초보자**: 복잡한 개념 없이 바로 개발하고 싶은 경우
</CalloutBox>

## 각 도구의 특별한 기능들

### CRA의 특별한 기능

**PWA 지원**: Progressive Web App을 쉽게 만들 수 있는 템플릿 제공

```bash
npx create-react-app my-pwa --template pwa
```

**TypeScript 지원**: TypeScript 템플릿으로 바로 시작

```bash
npx create-react-app my-ts-app --template typescript
```

**Eject 기능**: 필요시 모든 설정을 노출시켜 커스터마이징 가능

```bash
npm run eject  # 주의: 되돌릴 수 없음
```
<br/>
### Vite의 특별한 기능

**플러그인 생태계**: 다양한 플러그인으로 기능 확장

```js
// vite.config.js
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    port: 3000,
    open: true
  }
})
```

**환경별 설정**: 개발/운영 환경별로 다른 설정 적용

**CSS 전처리기**: Sass, Less, Stylus 등을 설정 없이 사용
<br/>
### Parcel의 특별한 기능

**자동 설치**: 필요한 패키지를 자동으로 설치

```html
<!-- 이렇게 import하면 자동으로 sass 설치됨 -->
<link rel="stylesheet" href="./styles.scss">
```

**다양한 파일 형식**: 이미지, 폰트, 비디오 등을 import로 바로 사용

**Code Splitting**: 자동으로 코드를 분할해서 로딩 최적화

## 마이그레이션: 다른 도구로 갈아타기

### CRA에서 Vite로 마이그레이션

기존 CRA 프로젝트를 Vite로 옮기는 것은 생각보다 간단합니다:

1. **Vite 설치**
```bash
npm install vite @vitejs/plugin-react --save-dev
```

2. **vite.config.js 생성**
```js
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    port: 3000
  }
})
```

3. **package.json 수정**
```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
```

4. **index.html 이동**: public/index.html을 루트로 이동하고 스크립트 태그 추가

### 마이그레이션 시 주의사항

<CalloutBox type="warning" title="마이그레이션 주의사항">
**환경변수(.env 파일)**: REACT_APP_ → VITE_ 접두사 변경 필요, 또한 호출 시, import.meta.env로 호출해야함.

**Public 폴더**: 파일 참조 방식이 다를 수 있음

**빌드 출력**: 빌드 결과물 구조가 다를 수 있음

**플러그인**: CRA용 플러그인은 Vite용으로 교체 필요
</CalloutBox>

### 2025년 현재 트렌드

1. **CRA 사용률 감소**: 새 프로젝트에서는 거의 사용하지 않음
2. **Vite 급성장**: React 공식 문서도 Vite 추천
3. **메타 프레임워크 성장**: Next.js, Remix 등의 인기 증가
4. **번들러 없는 개발**: Deno, Bun 같은 새로운 런타임 등장
