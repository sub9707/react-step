## TypeScript란?

### Javascript를 잠깐 다시 봅시다.
JavaScript는 1995년 Netscape에서 10일만에 만들어낸 언어란 사실은 꽤 유명하죠?

이 때 당시, 이 언어의 목적은 간단했습니다.

> “밋밋한 웹 페이지에 동작을 추가해보자!”

버튼 클릭 이벤트, 폼 데이터 검증, 알림 띄우기 등 간단한 동작을 추가하기 위한 보조 언어에 불과했죠.

하지만 세월이 지나면서 상황이 달라졌습니다.

AJAX로 새로고침 없이 데이터를 가져올 수 있게 되면서, React·Vue 같은 프레임워크가 등장했고

몇 백 줄이면 충분하던 코드가 수만, 수십만 줄로 커져버렸습니다.


<br/>

### 드러나는 문제점들

코드가 커지고 복잡해지자, 그동안 가려져 있던 문제가 드러나기 시작했습니다.

Java나 C 언어와 같이 변수 선언 시 자료형을 미리 정해놓고 사용하는 언어에 비해, 

JavaScript는 let, const, var 키워드로 변수만 선언하면 자료형은 런타임에 자동으로 결정할 수 있죠.    

이 자유로움이 처음엔 편했지만, 규모가 커질수록 에러의 씨앗이 되며 곯머리를 앓기 시작했습니다.

몇가지 경우들을 살펴볼게요.

---
경우 1: 자료형 혼동
<br/>

```Javascript
function calculateTotal(price, tax) {
  return price + tax;
}

// 정상 사용 (숫자)
calculateTotal(1000, 3);  // 1003

// 실수로 문자열 전달
calculateTotal("1000", "3");  // "10003"
```

총액을 계산하는 간단한 함수입니다.

숫자가 아닌 문자열이 들어가면, 더하기(+) 연산은 수학적 덧셈이 아니라 문자열 합성으로 동작합니다.

즉, 1000 + 3이 아니라 "1000" + "3"이 되어 "10003"이 되어버리죠.

함수가 조금만 복잡해져도 이런 버그는 금방 숨어버립니다.
<br/>
---
경우 2: 오타도 오류가 아니다
<br/>
```js
const user = {
  name: "김철수",
  email: "kim@example.com",
  age: 25
};

function displayUser(user) {
  console.log(user.nmae);  // undefined
  // name이 아니라 nmae인 오타!
}
```
분명 name인데 nmae로 오타를 냈습니다.

하지만 JavaScript는 아무런 경고도 주지 않고 그저 undefined를 출력합니다.

개발자는 나중에 콘솔이나 로그에서 ‘이상한 undefined’를 발견하고 뒤늦게 원인을 찾아야 했죠.

---
예제 3: 대규모 프로젝트의 악몽
<br/>

작은 예제에서는 그럭저럭 버틸 수 있지만,

수십 명의 개발자가 함께 작업하는 대규모 프로젝트에서는 문제가 훨씬 커집니다.

- 누군가 함수 매개변수에 엉뚱한 타입을 넘겨도 에러 없이 동작

- 객체 구조가 살짝 바뀌었는데, 여러 파일에서 참조하는 코드가 다 깨짐

- 공용 유틸 함수의 리턴값이 변경되면, 관련 모듈 전부가 런타임에서 터지고 나서야 알게 됨

즉, 타입 안정성이 부족한 언어의 한계가 드러난 거죠.
